TL;DR

This thesis proposes an SSI-based credential management system (Pistis) on top of the concepts of VC and DID of W3C. The Pistis system is designed to run on Ethereum.

A user forms a credential including his claims and sends it to an issuer through the Pistis application. The issuer issues the corresponding Verifiable Credential and sends it to the user. When the user wants to get a service from a service provider, the user presents his Verifiable Credential to a verifier who also uses the Pistis system . Note that all the issued credentials are stored in the user’s mobile device locally.

This construction uses some centralized components like web servers, and doesn’t discuss the how to achieve Sybil resistance. It is mostly a reference about implementing W3C’s specifications for Decentralized Identities and Verifiable Credentials.

Introduction

This note assumes some familiarity with W3C’s specification of Decentralized Identifiers (DIDs) and Verifiable Credentials (VCs)

Building blocks

We first outline the main components that are used in Pistis.

  • DIDs and DID Documents. These are implemented following W3C’s specification.
  • VCs. Again, these are implemented following W3C’s spec.
  • Blockchain
  • Smart Contracts
    • Operation executor
    • Permission registry
    • Trusted contact management
    • Credential status registry
    • DID registry
    • MultiSig operations
    • Revocation
  • Actors (All need to have Ethereum key pairs)
    • User(s)
    • Issuer(s)
    • Verifier(s)

Architecture

The architecture of the Pistis credential management system is as follows. We have mobile applications, client dashboards, web servers, and a blockchain. The mobile applications and client dashboards communicate with a web server directly. The mobile application and the web server communicate with the blockchain through Infura, and the client dashboard communicates with the blockchain through Metamask.

DID management

Pistis achieves DID management by DID creation, DID reading and verification, DID update, and DID deactivation functionalities.

DID Creation

Any user who has a key pair for Ethereum can create a DID to use it in Pistis. Assume that a user has an Ethereum address 0xAB, then his DID will be of the form did:pistis:0xAB.

See W3C’s Decentralized Identifiers data model v1.0 for more information about DID’s.

DID Read and Verify

This is achieved through the DID Resolver which is a software component with an API designed to accept requests for DID lookups and execute the corresponding DID method to retrieve the authoritative DID Document.

The DID Document contains a list of addresses that are authorized to sign on behalf of that DID, thus allowing a verifier to check authenticity by applying the asymmetric keys algorithm and checking whether the signing key is amongst the ones who have authorization permissions.

DID update

It is done by updating the DID Registry Smart Contract that holds the mapping between DIDs and the relative addresses with their specific permission for that DID.

DID deactivate

It is achieved by revoking all addresses for a certain DID. This makes the DID unusable and non-retrievable from that point onward.

Processes

In this section, we describe the issuance, sharing, revocation, and entity resolution processes.

Issuance of Verifiable Credentials

  • Upon receiving a credential request, the web application which is composed of front-end and back-end:
    • generates the Verifiable Credential,
    • signs it with its own private key,
    • wraps it into a Verifiable Presentation (VP),
    • produces a JWT of type “Attestation”,
    • renders it as a QR code.
  • The mobile app (user) checks the credential it received. If it is valid,:
    • The mobile application verifies if the issuer is trusted by the user: this amounts to checking that the issuer’s DID is associated with a contact already stored in the list of trusted contacts of the application.
    • If not, the application asks the user if he wants to add the new issuer to the trusted contacts list with the information attached in the VC field of the Verifiable Credential.
    • The Verifiable Credential is stored in the application's local storage.

Presenting Verifiable Credentials

Presenting VC happens when some service requires the user to be eligible for that service. In order to do so, the user must prove he matches the service requirements.

  • A QR code is shown on the screen for the user to scan using the mobile application. The QR code carries a JWT (JSON Web Token) which is a presentation request.
  • If the user accepts, the requested Verifiable Credentials are packed inside a Verifiable Presentation (VP) which is signed by the user and sent to the callback URL inside a JWT.
  • Then, the Verifiable Presentation needs to be verified,
    • first, for the verifiable presentation
    • second, for each of the credentials inside the presentation.
  • After that, the verifier needs to perform verification about the issuer of the Verifiable Presentation and Credential. These are done using Trusted Contact Management.

Revoke Verifiable Credentials

  • A revocation list is held in a smart contract (the “Credential Status List” smart contract).
  • In order to revoke Verifiable Credentials and verify their status, the W3C standard is applied. This standard proposes to insert a pointer to a claim revocation list inside the VC itself.
  • This pointer points to the Credential Status List smart contract.
  • A function accepts as parameters the credential issuer, the credential id, the status, and the status reason. It then updates the smart contract storage.

Entity Resolution (Trusted Contacts)

  • Just the way it happens with mobile phones in which you choose whose number belongs to depending on your knowledge and on who you trust. In the same way, in Pistis, the user has a Trusted Contacts List (TC List) which simply associates an Entity to a DID.

Exemplary implementation

The thesis provides different Proof of Concepts for Pistis. In this section, we describe one of them. The proof-of-concept aims to manage Malta’s Healthcare patients’ records. The example assumes an application which is called MyHealth.

Who is involved?

As usual, we have three actors here: the issuer (The Electronic-ID provider and hospital), the verifier (the doctor and the MyHealth app), and the user (Andrea).

User onboarding

Andrea creates a DID and forms a credential including his attributes. He sends his credential to the E-Id provider for issuance. The E-Id provider issues a verifiable credential that is private to Andrea. He can reach the issued credential through the application.

Book x-ray scan

For example, Andrea sees his doctor, and the doctor wants an x-ray scan from Andrea. The doctor also logs in to the MyHealth application and wants Andrea to share his E-Id by scanning the QR on the screen. The app verifies the identity of Andrea. The doctor can now book an x-ray scan for him. The MyHealth application associates the time slot that they desire with Andrea’s DID.

Perform x-ray scan

When Andrea goes to the hospital for the booked x-ray scan, he simply proves his identity by scanning the QR with his mobile application. He doesn’t share any personal information with anyone.

Receive scan by e-mail

When the results of the x-ray scan are ready, an e-mail is sent to Andrea. These results are verifiable credentials. Andrea downloads those credentials (results). All the credentials are stored locally.

Show the scan to the doctor

Andrea goes to the doctor, and the doctor simply shows a QR to Andrea through his computer's screen using the MyHealth application. Andrea scans the QR and chooses which credentials he wants to present. After Andrea’s confirmation, the x-ray scan results (credentials) are presented to the doctor through the MyHealth system.
